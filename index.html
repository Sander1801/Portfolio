Development

Last Resort is a school project we've been working on all year, four days a week. I mainly worked on artificial intelligence and quality assurance/management. Artificial intelligence was my main role, and quality management was a supporting role that I mainly did in the first three of the four blocks.

Many things went wrong in production and development in our team, but it was a learning experience with such a large team.

In the first block, we started with the concept phase. During this phase, we researched all kinds of techniques, design ideas, which engine to use (Unreal Engine 4), and the like. Together with one other AI programmer, I mainly researched AI techniques, performance, optimization, tools, and the like. For QA, I made a document related to the procedures and pipelines. You can think of making a game design document, technical design document, code reviews, Perforce shelves, how to report bugs, test cases, et cetera. In terms of AI, I've worked on benchmarking the AI, optimizing AI characters, wall climbing, and doing most of the research. Regarding research, I looked into AI decision-making, tactical pathfinding, overwhelming AI, movement, sensors, squad behavior, custom navigation, animation (blending and composite), kinematics, architecture (HTN and squad system), debugging & logging.

In the second block, we started with the pre-production phase. During this phase, I started with a presentation about the QA procedures and pipelines. I set up multiple things:

1. GameAnalytics.

2. Google Analytics connected with the Steam page.

3. Sentry.

4. Jenkins (including uploading to Steam).

5. A separate project for automation tests in Unreal Engine 4.

6. A list of analytics we should/want to track.

7. Custom fields in Jira for bug tracking (our team used Jira in the first two blocks, but after that, our team decided to use Monday).

8. A Python script that you can execute in Unreal Engine 4 to see if you have any blueprint error or warning.

I worked on multiple AI systems that I wrote in C++ (including UML diagrams and a few automation tests):

1. Squad and commander system.

2. Token system.

3. Custom movement component.

4. Flocking system.

5. Custom AI controller.

6. Utility system useable in a behavior tree (unfortunately, we had to scrap the HTN planner as it took too much time for someone to develop). As you can see below, a few utility axes that are being used for targeting:

7. Use the visual logger to draw AI information.

8. Animation update rate optimization component.

9. Custom navigation grid (in-progress, but mostly completed)

    - A* pathfinding.

    - JSON saver and loader (saves the navigation grid when adjusting the volume in the editor and loads the grid when you start the game, meaning that end-users don't have to build a navigation grid when they start a game).

    - Of course, the navigation grid itself. It also considers curves. Unfortunately, it doesn't support small stairs as a navigation grid isn't the best choice for that.



Besides that, I made sure that people kept following the pipelines, and I did several test cases and user acceptance tests to check the quality of the product.

In the third block, we started with the production phase. During this phase, I improved Jenkins and so on. I had more meetings with the producers to give them feedback on what is relevant, but unfortunately, the product management and planning didn't go well in our team. Furthermore, I arranged a heatmap tool from a different team and figured out how to integrate it into our project. I also did more test cases and kept track of the bug list, including setting the priority of bugs.

Regarding AI, I had to improve the utility system as we didn't go for the HTN planner anymore. After that, I implemented wall climbing using the navigation grid and made the navigation grid a lot better. Unfortunately, we decided to scrap the wall-climbing feature later as we were far behind with our game, including AI. Fortunately, we got a new AI programmer that block, but it took some time to get him on the same level as the programmer was new to AI.

Later, I worked on AI targeting using the utility and squad system. As of last, I helped a lot with solving bugs.

At the end of the block, we were still very behind schedule, and we had to work hard in the last block.

During the last block, I decided to distance myself mostly from QA management. I still kept track of the bug list and prioritized the bugs. I also made sure that the project builds kept working on Jenkins. I mainly distanced myself from my supporting role because professionalism was hard to find in our team, and my motivation also started to decrease.

Beyond that, I've worked a lot on bug fixing, optimizing the AI ​​and navigation mesh, helping with lighting issues, and improving the AI ​​(crowd movement, better targeting and attacking, et cetera).

Bongo Beater

School gave us a number of requirements that our game had to meet. Together with an artist, we came up with the Bongo Beater concept.

I started researching how to recognize beats in a song. Our initial plan was to automate everything so that designers don't have to create levels by hand, but since we were short on time, we decided to make a level editor just to be sure. I then worked on a tool that allowed designers to easily place hit points and hold points (think of the game Guitar Hero). I did this by creating custom C++ code that uses the level sequencer in Unreal Engine 4.

There are five lines in the level, and you can set the keys per line. Each line has two sublines. The first subline is for the hit points, and the second line is for the hold points.

Another important feature I added was the procedural environment in the background. So, you have an ellipsoid with randomly generated objects that scale up and down based on the sound.

Ray Tracer

As described on the main page, we had to make a ray tracer from scratch in C++. In addition, we also had to create our own vector and matrix class. Before I started programming, I analyzed the Turner Whitted reference photo (the first photo we had to remake) and researched all kinds of techniques, algorithms, and the like. In the end, I remade both reference photos (Turner Whitted and a circle of boxes, see main page) and applied all kinds of extra features, such as soft shadow, ambient occlusion, hollow sphere, multithreading, et cetera.

Vectors and matrices are widely used in game development, and by creating your own class, you actually learn the basics of game development/mathematics, such as the dot product, cross product, inverse, transpose, and the like.

For example, to calculate the inverse of a matrix, I applied three methods:

1. Hardcoded the computation (four by four matrix).

2. Analytical solution.

3. Gauss Jordan algorithm.

We had to create our own unit tests to ensure that certain algorithms work:

To make the program faster, I researched acceleration data structures, multithreading, and cache coherence. And, of course, I applied them. Unfortunately, I didn't have time for SIMD.

I chose to apply a bounding volume hierarchy as it is one of the better methods. Ideally, I wanted to implement a k-dimensional tree, but due to lack of time, I couldn't. When making all these techniques, it is also crucial to understand the Big O Notation. The render and build complexity has therefore also been taken into consideration.
